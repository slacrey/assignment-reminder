# 催孩子写作业系统 MVP 设计

日期：2026-06-10

## 背景

第一版要验证一个最小闭环：家长添加孩子并绑定 QQ，创建作业并设置提醒时间，到点后系统自动生成一条 QQ 提醒记录。MVP 暂不接入真实 QQ 发送能力，而是使用模拟发送日志确认调度和消息内容是否正确。

## 已确认决策

- 操作入口：本地 Web 管理页。
- 运行方式：本地单机运行，浏览器访问。
- QQ 发送方式：第一版模拟发送，只记录日志。
- 数据存储：本地 SQLite。
- 核心目标：跑通孩子管理、作业创建、到点提醒、提醒日志四件事。

## 目标

1. 家长可以添加孩子，填写孩子姓名和 QQ 号。
2. 家长可以为某个孩子创建作业，设置提醒时间。
3. 后端在提醒时间到达后自动处理待提醒作业。
4. 系统生成模拟 QQ 消息并写入提醒日志。
5. 前端能展示孩子、作业状态、提醒日志。

## 非目标

- 不做真实 QQ 登录和消息发送。
- 不做家长账号、登录鉴权和多用户隔离。
- 不做公网部署、多设备同步或云端数据。
- 不做复杂重复提醒、催促升级、作业完成确认。
- 不处理跨时区，本地系统时区即为 MVP 时区。

## 架构

MVP 是一个轻量全栈本地 Web 应用：

- Web 前端：提供家长管理界面，负责表单录入、列表展示和状态刷新。
- 后端 API：保存孩子、作业和提醒日志，向前端提供 REST 接口。
- SQLite 数据库：以本地文件保存数据，不依赖外部数据库服务。
- 后台定时器：随服务启动，每 30 秒扫描到点且未提醒的作业。
- 模拟 QQ 发送器：生成消息内容并写提醒日志，后续可替换为真实 QQ 适配器。

第一版保持单进程形态，前端通过 API 轮询获取最新作业状态和提醒日志。

## 数据模型

### children

保存孩子信息。

- `id`
- `name`
- `qq_number`
- `created_at`
- `updated_at`

约束：`name` 必填，`qq_number` 必填且是数字字符串。

### assignments

保存作业和提醒状态。

- `id`
- `child_id`
- `title`
- `description`
- `remind_at`
- `status`
- `created_at`
- `updated_at`

`status` 第一版使用：

- `pending`：待提醒。
- `reminded`：已提醒。
- `cancelled`：已取消。

创建作业时必须选择已有孩子，提醒时间必须是未来时间。

### reminder_logs

保存每次模拟发送结果。

- `id`
- `assignment_id`
- `child_id`
- `target_qq`
- `message`
- `scheduled_at`
- `sent_at`
- `status`
- `error_message`
- `created_at`

`status` 第一版使用 `success` 和 `failed`。

## 用户界面

首页就是管理工作台，不做营销页。页面分为三个区域：

1. 孩子管理
   - 新增孩子：姓名、QQ 号。
   - 孩子列表：姓名、QQ、作业数量、最近提醒时间。
   - 支持编辑孩子信息。
   - 删除孩子第一版可暂不提供，或在有作业时禁止删除。

2. 创建作业
   - 表单字段：孩子、作业标题、作业说明、提醒时间。
   - 作业列表：按提醒时间排序，展示待提醒、已提醒、已取消状态。
   - 待提醒作业可以取消；已提醒作业不能取消。

3. 提醒日志
   - 展示实际处理时间、孩子、QQ、作业标题、消息内容、状态。
   - 前端每 10-15 秒刷新作业和日志，便于看到刚触发的提醒。

整体风格应偏家长工作台：清楚、直接、可扫描，避免过度装饰。

## API

第一版 API：

- `GET /api/children`
- `POST /api/children`
- `PATCH /api/children/:id`
- `GET /api/assignments`
- `POST /api/assignments`
- `PATCH /api/assignments/:id/cancel`
- `GET /api/reminder-logs`

所有写接口都做服务端校验。前端展示后端返回的字段级错误或通用错误。

## 定时任务

服务启动后开启后台扫描器，每 30 秒执行一次：

1. 查询 `status = 'pending'` 且 `remind_at <= now` 的作业。
2. 对每条作业生成模拟 QQ 消息。
3. 在数据库事务中写入 `reminder_logs`。
4. 将作业状态更新为 `reminded`。

为避免重复提醒，更新作业状态时应带上 `status = 'pending'` 条件。只有状态更新成功才写成功日志。

模拟消息格式：

```text
作业提醒：{孩子姓名}，现在该写作业了。
作业：{标题}
说明：{说明}
```

如果说明为空，可以省略说明行。

## 错误处理

- 表单校验失败返回 `400`。
- 孩子或作业不存在返回 `404`。
- 取消已提醒作业返回明确错误：“已提醒的作业不能取消”。
- 定时扫描处理单条作业失败时写失败日志，不阻塞其他作业。
- 后端异常写入服务日志，前端展示简短通用错误。

## 测试策略

后端测试：

- 添加孩子成功和失败。
- 创建未来提醒作业成功。
- 创建作业时孩子不存在会失败。
- 到点扫描后作业只提醒一次。
- 取消作业后不会被提醒。
- 提醒日志记录消息、QQ、计划时间和实际时间。

前端测试或手动验收：

- 可以添加孩子并在列表看到。
- 可以创建作业并看到 `pending` 状态。
- 到提醒时间后，状态变为 `reminded`，日志出现模拟发送记录。
- 取消待提醒作业后不会生成提醒日志。

## 后续扩展

真实 QQ 接入时新增 QQ adapter 即可：

- 模拟发送器接口保持稳定。
- 真实发送器负责调用 OneBot、NapCat、Lagrange 或其他 QQ 方案。
- `reminder_logs` 继续记录真实发送结果和失败原因。

后续还可以增加重复提醒、完成确认、家长账号、移动端适配和部署配置。
